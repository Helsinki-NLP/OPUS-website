LetsMT's System Components

The LetsMT system consists of various components each serving its own purpose (like data storage, task execution, etc.). All components can be installed on the same machine but can also be distributed across several machines on a network for scalability. The following shows an overview of the system and describes each component, its role and services it consists of.

RR Server

Only one DB server exists for the system.

Roles

  • Holds and servers the meta data and group/permissino DB for the whole system.
  • Offers web services to serve meta and group data
  • Resolves storage location adresses via meta DB

Services

  • Meta DB, Group DB
  • WebServices
  • SGE submit host?

Storage Server

Several storage server can exist in the system to scale space and improve accessibility.

Roles

  • Holds a storage backend and serves the data in it
  • Offers web services to serve the storage data
  • Submits batch jobs to SGE master

Services

  • Storage Backend
  • WebServices
  • SGE submit host

Frontend Client

(Several) front end clients can use the web services to talk to the DB and Storage Servers

Roles

  • Uploads user data to the storage servers
  • Starts actions
  • writes and requests meta- and group data on DB server

Services

  • WebServices

SGE Master

One standart SGE master with a queue 'letsmt' to execute import, conversion, alignment and other jobs on the SGE execution clients

SGE Client

(Several) SGE clients execute the batch jobs

Roles

  • Execute commands defined in job description files that they fetch from the storage servers.
  • Resulting files and logs are uploaded back to the storage server
  • Meta data is written and updated on the meta DB during the whole job execution to inform about status and progress

Services

  • WebServices
  • SGE execution client
