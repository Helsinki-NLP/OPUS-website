Basic Concepts of the LetsMT! repository

Here is a list of very basic concepts of the LetsMT repository:

Corpus = Collection of text documents in XML

  • several (many!) documents from one domain & provider
  • may contain data in several languages
  • content & meta-data

Parallel Corpus = Collection of parallel documents

  • a corpus with (many!) documents in various languages
  • pairwise sentence alignments between corresponding documents

Parallel documents in LetsMT are documents that

  • exist in several languages
  • are completely translated

A pair of aligned documents is usally called a bitext.

Shared resources

Each corpus is a unique resource with a unique name. However, a corpus can be shared among several users and there might be several versions of that corpus. This means a corpus can grow, or shrink or change in various ways (include new documents, included documents may change ...). In order to provide a repository with data integrity for all users we introduce branching as the main mechanism for data sharing.

Basically a shared resource (corpus) is handled as a branch of the original corpus and may develop independent of the original version (this is specific to our version-controlled solution for storing repositories). In fact, the original corpus is also a branch owned by the initial provider of that corpus. To sum up, this means that a user who owns or shares a corpus will have his/her own branch and only this user may modify files and permissions in that branch. The name of that branch is simply the unique user name!

Internally the structure of the LetsMT repository looks like this (example): 

storage/CORPUS_1
storage/CORPUS_1/USER_1
storage/CORPUS_1/USER_1/xml
....
storage/CORPUS_1/USER_2
storage/CORPUS_1/USER_2/xml
...
storage/CORPUS_2
storage/CORPUS_2/USER_6
...

Obviously, users can only share resources from others for which they have permissions. These resources can be "public" or "shared" via group membership (each user is a member of one or several groups). A branch is created at the time a user decides to share a new resource. Updating this branch with information from other branches (merging) is not supported. This means that each branch will develop independent of all others from the time of creation.

Externally branches will not be visible for LetsMT users. Each user will be able to browse through his/her own resources (including shared resources) but also through available resources provided by others. However, before using resources by others he/she has to select those resources to be included in his/her own repository/list of resources (which internally means that a new branch is created from that resource).

The big advantage of branching data integrity: users do not depend on others and their modifications. The disadvantage is that users who share a resource do not have access to extensions/improvements of the original corpus. However, a mechanism that informs users who share a resource about new revisions could be implemented. These users can then decide to drop their own branch and to creata a new copy (branch) of the fresh revision.

A merging function could possibly also be implemented but this would require proper conflict handling via appropriate interfaces.
