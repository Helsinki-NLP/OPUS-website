Basic Functionalities

This is a list of basic tasks that should be supported by the LetsMT platform (this list is not complete or finalized in any way)

1) Uploading resources

(only registered users? I guess so ....)

  • user creates a new resource (corpus) 
  Input:
   - resource name (= corpus name = repository name = source of the data)
   - meta-data for the new resource
     * domain (& subdomain?)
     * data provider (= user, no need to fill in ....)
     * data owner (of original source) --> can be difficult for provider!
     (check discussion with partners, deliverables, ...)
  Action:
   - system check if resource (with that name) already exists
     YES --> ask user if he wants to create a branch (if permissions allow)
     NO ---> create empty resource
   - initialize (database entry, etc ...)
  • important: careful name checking for files and corpora (avoid problematic names)
    • no spaces
    • only ascii characters
    • remove other characters automatically?
    • warn/inform users about this restriction!
  • user uploads new TMX file: 
  Input:
   - resource (out of user-specific list)
   - source language
   - target language 
     (should we support multiple target languages? take lang-IDs
      directly from file? don't bother users?)
   - TMX (plain, zip, gz)
  Action:
   - warning if file exists already (update?)
   - upload to repository (original format), (file name normalization?)
   - add entry in meta-info database
     (initial status: raw)
     * additional meta-info: upload date, size, type, ...?
   - schedule validation & pre-processing job(s)
     (format specific, language specific)
  • user uploads new monolingual data file
    • same as above but no target language, different pre-processing jobs

2) pre-processing jobs

  • TMX validation & pre-processing jobs (off-line through job server)

Conversion jobs: 

  1. extraction (if TMX is compressed)
  2. validation: valid TMX? matching language tags? 
                --> add status "valid"
  3. conversion:
    a) create new XML file in corpus for each language
      --> use 2-letter language codes (ISO639-1) as subdirectory (+ region code if necessary?)
      --> new generic file name to avoid problems with strange characters (?)
    b) create sentence alignment file for each language pair
       (symmetric alignment! --> each pair only once!)
    c) extract aligned "sentences" (tuv's) from TMX
      --> remove (?) inline markup, add sentence markup with unique ID
      --> add sentences to appropriate corpus files
      --> add sentence alignment to appropriate sentence alignment file
      --> add status "converted"
    d) commit & create pre-processing jobs
      --> check-in all new files
      --> for each corpus file: create language-specific pre-processing jobs
      --> for each sentence alignment file: create initialization job

Pre-processing jobs: 

  1. checkout "converted" files
  2. language-specific or generic tokenization (no sentence splitting)
      (--> other stuff? tagging etc?)
  3. commit final version (sentence align file should link to those)
      --> add status "ready" for that file
      --> store information about pre-processing (what has been done?)
          (specific version of a specific Makefile?)
  4. update global information about corpus
      --> new language file is ready
      --> new statistics

Alignment initialization job: 

  1. wait until all pre-processing jobs are finished 
     (usually two -- one for source and one for target)
  2. update global information about corpus (if pre-processing succeeds)
      --> new global sentence alignment files (including the new resource)
      --> compute new statistics
  3. add information to meta-info database (or generate error report)
      --> add new version to meta-info database
      --> new sentence alignment file
      --> new corpus files (for each language)
      --> some statistics about the new data
  • pre-processing monolingual data 
    1. format specific validation and text extraction
    2. language-specific pre-processing (see above but also sentence splitting)
  • unaligned parallel documents 
   1. format-specific vaildation & conversion (like for monolingual data)
   2. language-specific pre-processing (like for monolingual data) 
   3. automatic sentence alignment
      --> status: "verification required"
   4. after verification: add as new parallel resource in corpus

3) Resources management

 * share resource (or remove permissions)
  --> group definition, group changes, ...
 * change pre-processing
  --> select different tokenizers, taggers, whatever we support ...
 * delete file from corpus
 * remove resource (do we allow that?)
 * later: on-line alignment ...

4) Use resources for SMT

 * browse/search through repository (using meta-info database)

 * select parallel training data
  --> select particular corpus (one language pair)
  --> select all shared data for a specific language pair
  --> select all shared data from a specific domain
  --> combine several selections

 * select monolingual data
  --> same as above but for one language only

 * download personal SMT model (from repository)

 * specify & submit a job
  --> using selected data
  --> usually: use standard settings (could include baseline data?)
  --> advanced: decide certain training parameters

 * job management
  --> inspect status of personal training jobs
  --> cancel running/queued jobs
  --> restart canceled jobs
